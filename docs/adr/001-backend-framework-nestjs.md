# ADR-001: Використання NestJS як backend framework

## Статус

Прийнято

## Контекст

Проєкт потребує backend framework для розробки API, агентів для обробки подій з черг, та інтеграцій з зовнішніми
сервісами (GitHub API, AI Provider API).

Ключові вимоги:

- Підтримка REST API для Dashboard
- Можливість створення background workers для агентів
- Інтеграція із чергами повідомлень (Redis)
- Чітка архітектура для 4 агентів + множинних модулів інтеграцій
- Простота тестування
- TypeScript підтримка

Розглянуті альтернативи:

- **Express.js** — мінімалістичний, але вимагає самостійної організації архітектури
- **Fastify** — швидкий, але unopinionated (треба самому структурувати)
- **NestJS** — opinionated framework з чіткою архітектурою

## Рішення

Використовуємо **NestJS** як основний backend framework.

## Обґрунтування

### Переваги NestJS для проєкту:

1. **Модульна архітектура**
   - Кожен агент (Architect, Workflow, Code Review, Documentation) — окремий модуль
   - Кожна інтеграція (GitHub, AI Provider, Database) — окремий модуль
   - Чітке розділення відповідальностей

2. **Dependency Injection**
   - Автоматичне управління залежностями між модулями
   - Простота тестування (легко мокати залежності)
   - Уникнення глобального стану

3. **Вбудована інтеграція з BullMQ**
   - `@nestjs/bull` пакет для роботи із чергами
   - Декларативні процесори подій через декораторів
   - Автоматичний retry та DLQ

4. **Декларативна валідація**
   - `class-validator` для валідації DTO
   - Автоматична генерація помилок валідації
   - TypeScript types + runtime валідація

5. **Структура "з коробки"**
   - Зрозуміло, де розміщувати код
   - Consistency у всіх модулях
   - Легше onboarding нових розробників

6. **Екосистема**
   - Офіційні інтеграції з Prisma, BullMQ, WebSockets
   - Велика спільнота
   - Багато прикладів та best practices

7. **Тестування**
   - Вбудовані utilities для unit та integration тестів
   - DI робить моки тривіальними

### Недоліки:

- Більша крива навчання (DI, декоратори, модулі)
- Більше boilerplate коду
- Трохи нижча performance порівняно з Fastify (але вирішується через Fastify adapter)

### Чому не інші варіанти:

**Express.js:**

- Відсутність структури може призвести до "неструктурованого коду"
- Треба самому імплементувати DI
- Важче організувати код для складного проєкту з 10+ модулями

**Fastify (без NestJS):**

- Unopinionated — треба самому продумувати архітектуру
- Немає вбудованої інтеграції з BullMQ
- Важче підтримувати при рості проєкту

## Наслідки

### Позитивні:

- Чітка структура проєкту із самого початку
- Простота додавання нових модулів/агентів
- Високий рівень тестованості коду
- Consistency у всьому проєкті

### Негативні:

- Потрібен час на вивчення основних концепцій NestJS
- Більше коду для простих речей (controllers, modules, decorators)

### Нейтральні:

- Команда повинна дотримуватись NestJS conventions
- Performance можна покращити через Fastify adapter (див. ADR-002)

## Примітки

- Використовуємо NestJS версії 10.x (LTS)
- Дотримуємось офіційних best practices з документації
- Кожен агент оформляємо як окремий модуль
