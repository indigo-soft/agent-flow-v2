# Модулі системи

Цей документ описує основні модулі системи **agent-flow-v2**, їх відповідальність, вхідні та вихідні дані, потоки та
функціональність.

## 1. Dashboard Module (Frontend) — Shared Component

**Тип:** Shared компонент (`components/dashboard/`)

**Відповідальність**: Інтерфейс користувача для взаємодії з системою

**Входи**:

- Дані від користувача (бесіди, зауваження, натискання кнопок)
- Дані з API Gateway (чернетки, завдання, статуси)

**Виходи**:

- HTTP-запити до API Gateway
- Відображення UI (форми, канбан, чернетки)

**Потоки**:

- Створення чернетки плану:
    - Користувач вводить текст бесіди → натискає "Обробити" → Dashboard відправляє POST до API Gateway → отримує
      чернетку → відображає для перевірки

- Додавання у роботу:
    - Користувач натискає "Додати у роботу" → Dashboard відправляє POST до API Gateway → отримує підтвердження

- Переробка чернетки:
    - Користувач вводить зауваження → натискає "Переробити" → Dashboard відправляє PUT до API Gateway → отримує оновлену
      чернетку

- Управління завданнями:
    - Користувач натискає кнопки ("Запланувати", "Взяти у роботу", "Зробити ревʼю", "Повторне ревʼю") → Dashboard
      відправляє PATCH до API Gateway → отримує оновлений список завдань

- Перегляд канбану:
    - Dashboard запитує GET до API Gateway → отримує список завдань → відображає у стовпчиках за статусами

**Функціональність**:

- Форма "Нова бесіда/інформація" з текстовим полем та кнопкою "Обробити"
- Відображення чернетки плану з можливістю редагування
- Поле для зауважень та кнопки "Додати у роботу" / "Переробити"
- Канбан-дошка зі стовпчиками: "Нові завдання", "Наступні завдання", "У роботі", "На доопрацюванні", "Виконано"
- Кнопки біля кожного завдання: "Запланувати", "Взяти у роботу", "Зробити ревʼю", "Повторне ревʼю"

## 2. API Gateway Module — Shared Component

**Тип:** Shared компонент (`components/api/`)

**Відповідальність**: Маршрутизація запитів та валідація даних

**Входи**:

- HTTP-запити від Dashboard
- HTTP-відповіді від внутрішніх сервісів

**Виходи**:

- Виклики до внутрішніх сервісів (Database Service, агентів)
- HTTP-відповіді до Dashboard

**Потоки**:

- POST /conversations/analyze:
    - Приймає текст бесіди → валідує → викликає Architect Agent → повертає чернетку

- PUT /drafts/{id}:
    - Приймає зауваження до чернетки → валідує → викликає Architect Agent → повертає оновлену чернетку

- POST /drafts/{id}/commit:
    - Приймає ID чернетки → викликає Architect Agent для формування завдань → викликає Database Service для збереження →
      повертає підтвердження

- GET /tasks:
    - Викликає Database Service → повертає список завдань

- PATCH /tasks/{id}/status:
    - Приймає новий статус → валідує → викликає Database Service → публікує подію у Event Queue (якщо потрібно) →
      повертає оновлене завдання

**Функціональність**:

- Валідація формату даних
- Аутентифікація/авторизація (якщо потрібна)
- Маршрутизація до потрібних сервісів
- Обробка помилок та формування відповідей
- Публікація подій при зміні статусів завдань

## 3. Database Service Module — Shared Component

**Тип:** Shared компонент (`components/database/`)

**Відповідальність**: Збереження та отримання даних з БД

**Входи**:

- Запити на CRUD операції від API Gateway та агентів
- Дані для збереження (чернетки, завдання, артефакти, статуси)

**Виходи**:

- Результати операцій (дані або підтвердження)
- Помилки БД

**Потоки**:

- Створення чернетки:
    - Приймає текст чернетки → зберігає у БД → повертає ID та дані чернетки

- Оновлення чернетки:
    - Приймає ID та оновлені дані → оновлює у БД → повертає оновлену чернетку

- Створення завдань:
    - Приймає ієрархію завдань від Architect Agent → зберігає у БД → прив'язує артефакти → повертає список створених
      завдань

- Отримання завдань:
    - Приймає фільтри (статус, ID) → запитує БД → повертає список завдань

- Оновлення статусу завдання:
    - Приймає ID завдання та новий статус → оновлює у БД → повертає оновлене завдання

**Функціональність**:

- CRUD для чернеток планів
- CRUD для завдань (епіки, задачі, підзадачі)
- Збереження критеріїв готовності
- Прив'язка артефактів до завдань
- Зміна статусів завдань
- Запити з фільтрацією та сортуванням

## 4. Event Queue Module — Shared Component

**Тип:** Shared компонент (`components/queue/`)

**Відповідальність**: Публікація подій та доставка їх підписникам

**Входи**:

- Події від API Gateway та агентів
- Підписки від агентів

**Виходи**:

- Доставка подій агентам-підписникам
- Підтвердження публікації

**Потоки**:

- Публікація події:
    - Приймає подію (тип, дані) → додає у чергу → повертає підтвердження

- Доставка події:
    - Читає подію з черги → знаходить підписників → доставляє подію → очікує підтвердження обробки → видаляє з черги (
      або повторює)

**Події у системі**:

- "Робота над завданням розпочата" (дані: ID завдання)
- "Зробити ревʼю" (дані: ID завдання)
- "Продовжити ревʼю" (дані: ID завдання, ID PR)
- "Повторне ревʼю" (дані: ID завдання, ID PR, коментарі, файли)
- "Завдання виконано" (дані: ID завдання, ID PR)
- "Оновити документацію" (дані: ID завдання, зміни)

**Функціональність**:

- Публікація подій
- Зберігання черги подій
- Доставка подій підписникам
- Retry механізм при помилках доставки
- Логування подій

## 5. GitHub Integration Module — Shared Component

**Тип:** Shared компонент (`components/github/`)

**Відповідальність**: Взаємодія з GitHub API

**Входи**:

- Запити від агентів (створити гілку, PR, ревʼю, мердж тощо)
- Дані для операцій (назви гілок, тексти ревʼю, коментарі)

**Виходи**:

- Результати операцій (дані гілок, PR, файлів)
- Помилки GitHub API

**Потоки**:

- Створення гілки:
    - Приймає назву гілку, base branch → викликає GitHub API → повертає дані гілку

- Створення PR:
    - Приймає назву PR, опис, source/target гілки → викликає GitHub API → повертає дані PR

- Отримання змінених файлів:
    - Приймає ID PR → викликає GitHub API → повертає список файлів з diff

- Публікація ревʼю:
    - Приймає ID PR, тип ревʼю (APPROVE/REQUEST_CHANGES), коментар → викликає GitHub API → повертає підтвердження

- Публікація коментарів:
    - Приймає ID PR, текст коментарів → викликає GitHub API → повертає підтвердження

- Отримання коментарів:
    - Приймає ID PR → викликає GitHub API → повертає список коментарів

- Мердж PR:
    - Приймає ID PR, метод мерджу → викликає GitHub API → повертає підтвердження

- Видалення гілку:
    - Приймає назву гілки → викликає GitHub API → повертає підтвердження

- Закриття PR:
    - Приймає ID PR → викликає GitHub API → повертає підтвердження

**Функціональність**:

- Обгортка над GitHub REST API
- Аутентифікація через токен
- Обробка rate limits
- Обробка помилок API
- Форматування запитів згідно визначеного формату

## 6. AI Provider Integration Module — Shared Component

**Тип:** Shared компонент (`components/ai-provider/`)

**Відповідальність**: Взаємодія з ШІ-провайдером

**Входи**:

- Промпти від агентів
- Конфігурація запиту (модель, температура тощо)

**Виходи**:

- Відповіді від ШІ
- Помилки ШІ-провайдера

**Потоки**:

- Аналіз бесіди (для Architect Agent):
    - Приймає текст бесіди → формує запит до ШІ → відправляє до ШІ → отримує структурований план → розбирає відповідь →
      повертає чернетку

- Формування завдань (для Architect Agent):
    - Приймає чернетку плану → формує запит до ШІ → відправляє до ШІ → отримує ієрархію завдань → опрацьовує відповідь →
      повертає список завдань

- Ревʼю коду (для Code Review Agent):
    - Приймає список файлів, diff, контекст завдання → формує запит до ШІ → відправляє до ШІ → отримує відповідь
      (APPROVED/REQUEST_CHANGES/QUESTIONS + текст) → аналізує відповідь → повертає результат ревʼю

**Функціональність**:

- Формування запитів до ШІ за шаблонами
- Відправка запитів до API ШІ-провайдера (OpenAI, Claude тощо)
- Парсинг відповідей у структуровані дані
- Обробка помилок провайдера
- Retry механізм
- Валідація формату відповідей

## 7. Architect Agent Module — Domain Module

**Тип:** Доменний модуль (`modules/architect-agent/`)

**Відповідальність**: Перетворення неструктурованої інформації у структуровані плани та завдання

**Входи**:

- Текст бесіди (від API Gateway)
- Зауваження до чернетки (від API Gateway)
- Чернетка плану для формування завдань (від API Gateway)

**Виходи**:

- Чернетка структурованого плану
- Ієрархія завдань з критеріями готовності
- Помилки обробки

**Потоки**:

- Аналіз бесіди:
    - Приймає текст → викликає AI Provider Integration → отримує чернетку плану → зберігає через Database Service →
      повертає чернетку до API Gateway

- Переробка чернетки:
    - Приймає ID чернетки та зауваження → завантажує чернетку через Database Service → викликає AI Provider Integration
      з зауваженнями → отримує оновлену чернетку → оновлює через Database Service → повертає оновлену чернетку

- Формування завдань:
    - Приймає ID чернетки → завантажує через Database Service → викликає AI Provider Integration для формування
      ієрархії → отримує список завдань → додає критерії готовності → прив'язує артефакти → зберігає через Database
      Service → повертає підтвердження

**Функціональність**:

- Аналіз неструктурованого тексту
- Виділення ключових рішень та завдань
- Формування чернетки плану
- Врахування зауважень користувача
- Створення ієрархії завдань (епік → задача → підзадача)
- Генерація критеріїв готовності для кожного завдання
- Прив'язка артефактів до завдань

## 8. Workflow Agent Module — Domain Module

**Тип:** Доменний модуль (`modules/workflow-agent/`)

**Відповідальність**: Управління lifecycle завдання від створення гілку до документації

**Входи**:

- Подія "Робота над завданням розпочата" (з Event Queue)
- Подія "Завдання виконано" (з Event Queue)

**Виходи**:

- Запити до GitHub Integration (створення гілки, мердж, видалення)
- Запити до Database Service (оновлення статусів)
- Подія "Оновити документацію" (до Event Queue)

**Потоки**:

- Початок роботи над завданням:
    - Отримує подію "Робота над завданням розпочата" → завантажує дані завдання через Database Service → формує назву
      гілки згідно формату → викликає GitHub Integration для створення гілки → оновлює завдання у Database Service (
      додає дані гілки)

- Завершення завдання:
    - Отримує подію "Завдання виконано" → завантажує дані завдання та PR через Database Service → викликає GitHub
      Integration для мерджу PR → викликає GitHub Integration для видалення гілки → викликає GitHub Integration для
      закриття PR → оновлює статус завдання на "Виконано" через Database Service → публікує подію "Оновити документацію"
      у Event Queue

**Функціональність**:

- Слухання подій з Event Queue
- Створення feature-гілок за форматом
- Мердж PR до основної гілки
- Видалення гілок після мерджу
- Закриття PR
- Оновлення статусів завдань
- Запуск процесу оновлення документації

## 9. Code Review Agent Module — Domain Module

**Тип:** Доменний модуль (`modules/code-review-agent/`)

**Відповідальність**: Організація процесу ревʼю коду через ШІ

**Входи**:

- Подія "Зробити ревʼю" (з Event Queue)
- Подія "Продовжити ревʼю" (з Event Queue)
- Подія "Повторне ревʼю" (з Event Queue)

**Виходи**:

- Запити до GitHub Integration (створення PR, отримання файлів, публікація ревʼю)
- Запити до AI Provider Integration (промпти для ревʼю)
- Запити до Database Service (оновлення статусів)
- Подія "Завдання виконано" (до Event Queue)
- Подія "Повторне ревʼю" (до Event Queue)

**Потоки**:

- Початкове ревʼю:
    - Отримує подію "Зробити ревʼю" → завантажує завдання через Database Service → викликає GitHub Integration для
      створення PR → отримує список змінених файлів через GitHub Integration → формує промпт з файлами та контекстом
      завдання → викликає AI Provider Integration → отримує відповідь (APPROVED/REQUEST_CHANGES/QUESTIONS) → якщо
      APPROVED: публікує ревʼю через GitHub Integration → публікує подію "Завдання виконано" → якщо REQUEST_CHANGES:
      публікує ревʼю через GitHub Integration → оновлює статус на "На доопрацюванні" через Database Service → якщо
      QUESTIONS: публікує коментарі через GitHub Integration → оновлює статус на "На доопрацюванні" через Database
      Service

- Запит повторного ревʼю:
    - Отримує подію "Продовжити ревʼю" → завантажує завдання та PR через Database Service → отримує коментарі через
      GitHub Integration → отримує змінені файли через GitHub Integration → формує запит з коментарями та файлами →
      публікує подію "Повторне ревʼю" із цим запитом в Event Queue

- Повторне ревʼю:
    - Отримує подію "Повторне ревʼю" → формує запит до ШІ з даними з події → викликає AI Provider Integration → отримує
      відповідь (APPROVED/REQUEST_CHANGES/QUESTIONS) → виконує ті ж дії, що й у початковому ревʼю

**Функціональність**:

- Слухання подій з Event Queue
- Створення PR за форматом
- Отримання списку змінених файлів та diff
- Формування запитів до ШІ для ШІ-ревʼю
- Парсинг відповідей від ШІ
- Публікація ревʼю на GitHub (approve/request changes)
- Публікація коментарів/питань до PR
- Управління циклом повторного ревʼю
- Оновлення статусів завдань

## 10. Documentation Agent Module — Domain Module

**Тип:** Доменний модуль (`modules/documentation-agent/`)

**Відповідальність**: Оновлення документації після виконання завдань

**Входи**:

- Подія "Оновити документацію" (з Event Queue)

**Виходи**:

- Запити до Database Service (отримання даних завдання)
- Запити до GitHub Integration (отримання змін, оновлення файлів документації)

**Потоки**:

- Оновлення документації:
    - Отримує подію "Оновити документацію" → завантажує дані завдання через Database Service → отримує зміни з PR через
      GitHub Integration → аналізує, які розділи документації потребують оновлення → формує оновлення документації →
      зберігає зміни через GitHub Integration (або через окремий коміт/PR згідно з вашим процесом)

**Функціональність**:

- Слухання подій з Event Queue
- Аналіз виконаного завдання
- Визначення розділів документації для оновлення
- Формування тексту оновлень
- Збереження оновленої документації
