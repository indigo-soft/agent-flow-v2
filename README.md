# AGENT-FLOW (AI-Powered Development Workflow Assistant)

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)

## Що це за проєкт

AI-Powered Development Workflow Assistant — це система автоматизації процесу розробки програмного забезпечення, яка
допомагає розробнику перетворювати неформальні обговорення та ідеї у структуровані завдання, а потім супроводжує весь
цикл їх виконання: від створення гілки у Git до ревʼю коду штучним інтелектом та оновлення документації.

## Що робить система

### 1. Перетворення обговорень у структуровані плани

Ви обговорюєте проєкт з колегою, ШІ-асистентом або просто робите нотатки про те, що потрібно зробити. Потім копіюєте
весь цей текст у систему, і вона:

- **Аналізує неструктуровану інформацію**
- **Виділяє ключові рішення та завдання**
- **Формує чіткий структурований план**
- **Дає вам можливість перевірити й виправити план перед тим, як він піде у роботу**

### 2. Автоматичне створення завдань

Після вашого підтвердження система:

- **Перетворює план в ієрархію завдань** (епіки, задачі, підзадачі)
- **Генерує критерії готовності** для кожного завдання
- **Прикріплює артефакти** (файли, посилання, скриншоти)
- **Додає завдання** у вашу робочу дошку

### 3. Канбан-управління завданнями

Система надає візуальну дошку завдань у стилі Kanban зі стовпчиками:

- **Нові завдання** — щойно створені, ще не заплановані
- **Наступні завдання** — заплановані до виконання
- **У роботі** — ви активно над ними працюєте
- **На доопрацюванні** — потребують виправлень після ревʼю
- **Виконано** — завершені та змерджені

Ви можете переміщати завдання простими кліками кнопок.

### 4. Автоматизація Git workflow

Коли ви берете завдання у роботу, система:

- **Автоматично створює feature-гілку** на GitHub з правильною назвою
- **Ви працюєте у цій гілці локально**, робите коміти та пушите зміни
- **Коли готові** — одна кнопка ініціює процес ревʼю

### 5. Автоматичне ревʼю коду через ШІ

Після вашого запиту на ревʼю система:

- **Створює Pull Request** на GitHub
- **Передає всі зміни у коді** штучному інтелекту
- **ШІ аналізує код** на предмет проблем, вразливостей, помилок
- **ШІ може дати одну з трьох відповідей**:
    - **APPROVED** — все добре, код можна мерджити
    - **REQUEST_CHANGES** — є проблеми, які треба виправити
    - **QUESTIONS — треба щось пояснити або уточнити**

### 6. Інтерактивний процес доопрацювання

Якщо ШІ запитує або вимагає змін:

- **Коментарі та питання** публікуються безпосередньо у Pull Request
- **Ви читаєте їх**, відповідаєте або виправляєте код
- **Запускаєте повторне ревʼю** однією кнопкою
- **Процес повторюється** до схвалення

### 7. Автоматичний мердж та закриття

Коли ШІ схвалює код:

- **Pull Request** автоматично мерджиться до основної гілки
- **Feature-гілка** видаляється
- **PR закривається**
- **Завдання переміщується** у "Виконано"

### 8. Оновлення документації

Після успішного завершення завдання:

- **Система аналізує** зроблені зміни
- **Автоматично оновлює** відповідні розділи документації
- **Документація** завжди актуальна

## Чим може бути корисним

#### Для індивідуальних розробників:

- ✅ **Структурування хаосу ідей** — перетворення безладних думок у чіткий план
- ✅ **Автоматизація рутини** — не треба вручну створювати гілки, PR, писати шаблони
- ✅ **Додатковий погляд на код** — ШІ може помітити те, що ви пропустили
- ✅ **Самодисципліна** — структурований workflow допомагає не забувати про важливі етапи
- ✅ **Актуальна документація** — документація оновлюється автоматично, а не коли "знайдеться час"

#### Для малих команд:

- ✅ **Єдиний workflow** — усі працюють за однією схемою
- ✅ **Прозорість процесу** — чітко видно, хто над чим працює
- ✅ **Зменшення навантаження на ревʼю** — ШІ виловлює базові помилки, людина дивиться на архітектуру
- ✅ **Швидший onboarding** — новачки одразу розуміють процес

#### Для навчання:

- ✅ **Вчитися best practices** — ШІ коментує код з поясненнями
- ✅ **Розуміння Git flow** — система демонструє правильний робочий процес
- ✅ **Формування навичок планування** — учитися розбивати завдання на підзадачі

#### Для MVP та стартапів:

- ✅ **Швидкий старт** — не треба налаштовувати складні CI/CD спочатку
- ✅ **Гнучкість** — легко адаптувати під свій процес
- ✅ **Економія часу** — автоматизація дає більше часу на продукт

## Основні переваги системи

- **Єдине місце для всього процесу** — від ідеї до виконання
- **Мінімум ручної роботи** — більшість рутини автоматизована
- **ШІ як асистент, не замінник** — ви контролюєте процес, ШІ допомагає
- **Прозорість та історія** — усе фіксується у GitHub та БД
- **Гнучкість** — можна адаптувати під свій workflow

## Швидкий старт

```bash
# Clone
git clone https://github.com/your-org/ai-workflow-assistant.git
cd ai-workflow-assistant

# Install
pnpm install

# Setup environment
cp .env.example .env
# Заповніть .env файл

# Run dev
pnpm dev
```

## Розробка

### Git Workflow

**⚠️ ВАЖЛИВО: Перш ніж почати розробку, прочитайте [Git Workflow Guide](docs/guides/git-workflow.md)**

**Коротко:**

1. **Створіть гілку від main:**
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/your-feature-name
   ```

2. **Робіть коміти за Conventional Commits:**
   ```bash
   git commit -m "feat(scope): description"
   ```

3. **Відкрийте Pull Request** коли готово

4. **Після мерджу видаліть гілку**

**Детальніше:** [docs/guides/git-workflow.md](docs/guides/git-workflow.md)

### Naming Conventions

**Гілки:**

- `feature/architect-agent-implementation`
- `fix/kanban-drag-drop-issue`
- `docs/api-documentation`

**Commits:**

- `feat(architect): add draft creation service`
- `fix(kanban): resolve mobile drag issue`
- `docs(readme): update setup instructions`

**Детальніше:** [docs/guides/naming-conventions.md](docs/guides/naming-conventions.md)

```bash
pnpm lint

pnpm format

pnpm typecheck

pnpm test
```
